# Phase 5: Subsurface Support

## Goal

Avoid simulating terrain as a hollow visual surface by adding shallow particles
below the visible sand layer.

## Inputs

- Local surface patch and normal from Phase 2.
- Scene scale.
- Desired support depth.
- Particle spacing and layer count.

## Current Method

The current implementation uses three explicit entities:

1. an interpolated surface cap generated from the splat terrain,
2. regular-grid subsurface particles below that cap,
3. a ground plane just below the lowest subsurface layer.

The raw splat points are used only to estimate the height field. They are not
written as the simulated/displayed surface entity, because the source splat PLY
can contain buried points below the visible sand surface. The exported
`particles_surface_mpm.ply` is now a regular interpolated cap.

The subsurface is no longer generated by cloning every surface point downward.
Instead, it builds regular XY grids in normalized MPM space, samples local
surface height from the splat-derived field, jitters each subsurface layer in XY,
then clamps the support particle height below the visible cap at that jittered
XY:

```text
surface_cap = regular_grid_xy + interpolated_splat_height
for each requested layer depth:
    xy_layer = regular_grid_xy + layer_shift + xy_noise
    z_surface = min(splat_height(xy_layer), visible_cap_height(xy_layer))
    particle = (xy_layer, z_surface - depth)
```

The current support depth is `0.2` source units. Each subsurface point receives
independent uniform XY jitter. The ground plane can be set from the lowest
subsurface particle plus a small offset; the current dense test uses
`--ground-subsurface-quantile 0.0 --ground-offset-below-subsurface 0.01`.

The current generated suite is:

```text
outputs/initial_multilayer_suite/
  summary.csv
  README.md
  spacing008/
  spacing004/
  spacing002/
  spacing001/
```

Each case writes:

```text
particles_initial_mpm.ply
particles_surface_mpm.ply
particles_subsurface_mpm.ply
ground_plane_metadata.json
initial_oblique.png
```

Current validated cap-only initial states:

```text
output                                             spacing_mpm  layers  surface_cap  subsurface  total
outputs/initial_robust_cap_only_surface_v7         0.01         4       21,165       82,563      103,728
outputs/initial_robust_cap_only_surface_spacing00025_v1
                                                   0.0025       4       60,778       209,757     270,535
```

Both validated outputs have `surface_entity_source = interpolated_cap`, zero
particles below the ground plane, and zero subsurface points above the
interpolated visible cap.

Do not generate `spacing00025` with automatic depth spacing casually: with both
XY spacing and layer spacing at `0.0025`, it is expected to create tens of
millions of particles for the current radius-1 crop. The validated
`spacing00025` case above is practical because it uses only four explicit layer
depths: `0.05,0.10,0.15,0.20`.

## Current Manual Splat-Slice Initializer

The preferred initializer now starts directly from the original EDGS splat PLY,
not from previously generated particle PLYs. The accepted visual slice is:

```text
source: ../EDGS/output/point_cloud/iteration_7000/point_cloud.ply
aligned splat z band: [-2.4, -2.1]
XY crop: centered 1.0 x 1.0 source-unit box
```

The visible surface is the colored splat slice. The subsurface is generated as
regular XY grids under a local high-quantile surface estimate, with per-layer XY
shift and random XY noise so particles are not vertical copied columns.

Current accepted single-case output:

```text
outputs/splat_surface_regular_grid_subsurface_1x1_depth0p2_spacing0p025_layer0p0125_noise1p5/
```

Its settings are:

```text
--box-size 1.0
--depth 0.2
--xy-spacing 0.025
--layer-spacing 0.0125
--xy-jitter 0.8
--noise-scalar 1.5
```

Validation for that case: 21,536 surface splats, 26,895 subsurface particles,
16 layers from depth 0.0125 to 0.2, and unique rounded XY coordinates for all
subsurface particles.

The 3x3x3 experiment matrix runner is:

```bash
conda run -n tsplat python scripts/run_splat_matrix_experiments.py \
  --layer-counts 8,16,24 \
  --layer-depths 0.1,0.2,0.3 \
  --particle-sizes 0.015,0.025,0.035
```

It writes each case under `outputs/splat_matrix_3x3x3/`, runs Genesis from the
prebuilt splat-slice particles, renders `solver_animation.mp4`, and records
`summary.csv`.

Do not run the full matrix while base physics is still being tuned. The active
single base case is:

```text
layers = 16
depth = 0.2
layer_spacing = 0.0125
particle_size = 0.0125
render fps = 60
latest output = outputs/base_earth_less_bouncy_layers16_depth0p2_ps0p0125/
```

The Genesis `sand_wheel.py` example does not use softer sand; it uses default
`gs.materials.MPM.Sand()` (`E=1e6`, `nu=0.2`, `rho=1000`,
`friction_angle=45`) with `SimOptions(dt=3e-3, substeps=10)`. Its main
differences are dynamic sand emission and coupled rigid contact settings. Next
base-case work should add solver substeps and set the ground rigid material to
`Rigid(needs_coup=True, coup_friction=0.2, coup_softness=0.0,
coup_restitution=0.0)`.

## PhysGaussian Reuse

PhysGaussian includes optional particle filling based on reconstructed density
and ray tests. That is useful for closed objects, but terrain is usually an open
surface. The terrain prototype should document a terrain-specific shallow-column
filling method as the first version, while keeping PhysGaussian filling as a
reference for inheriting visual attributes and particle volumes.

Filled subsurface particles should participate in simulation. They do not need
to render in the first version unless exposed by deformation.

## Deliverables

- Surface plus subsurface particle visualization.
- Local terrain volume export.
- MPM run with and without subsurface support.
- Comparison metrics for surface-only versus supported terrain.

## Success Criteria

- Particles do not collapse immediately.
- Downward displacement produces compaction and some lateral motion.
- Results look more plausible than surface-only MPM.
- The support depth and layer count are recorded with each run.

## Risks / Open Questions

- Too few layers behave like a shell; too many layers increase runtime.
- Incorrect normals can place support particles above or beside the terrain.
- Subsurface particles need material parameters even if they are not rendered.
